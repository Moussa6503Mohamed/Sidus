## Goal

Clerk owns authentication; Sidus Core owns authorization. No user-controlled
`actorId`/`reviewerId`. Audit identity (event actor, review reviewer) derives only from the
verified Clerk session subject. Role authorization from the verified `sidus_role` claim.

## Fail-open hardening (review follow-up)

Closed four fail-open gaps found in review. All are fail-closed now:

1. **Core issuer is mandatory.** `main.buildVerifier` (was `clerkVerifierFromEnv`) returns
   `(nil, "CLERK_JWT_ISSUER")` when `CLERK_JWT_ISSUER` is missing/blank/whitespace, so
   content-source routes do not mount. Previously a blank issuer built a verifier that skipped
   issuer pinning. `main` logs only the offending variable name — never a secret.
2. **Authorized parties never silently unrestricted.**
   - Core: `authorizedParties` uses `os.LookupEnv` to distinguish absent (→ dev default
     `http://localhost:3000` only) from present-but-blank (→ `(nil, false)` → routes do not
     mount). Previously a whitespace/commas-only value produced an empty allow-list, which the
     SDK `AuthorizedPartyHandler` treated as "accept any azp".
   - AI: `_authorized_parties_from_env` applies the same rule — absent → local default,
     present-but-blank → `ClerkConfigError`. `get_authenticator` returns `None` on any unsafe
     config; the `require_clerk_session` dependency then returns a generic **503**.
3. **AI issuer is mandatory.** `ClerkAuthenticator.__init__` raises `ValueError` when issuer is
   missing/blank, so a configured `CLERK_JWKS_URL` can no longer bypass issuer validation;
   verify options now always `require`/`verify_iss` the issuer. Missing issuer or blank azp →
   protected routes fail closed with `503 {"detail":"authentication is not configured"}` (no
   config detail or secret leaked). `/healthz` stays public.
4. **Legacy caller-identity fields rejected.** `contentsource.decodeStrict` sets
   `DisallowUnknownFields()` on every content-source body decode (create/update/approve/
   reject). Any unknown field — including legacy `actorId`/`reviewerId` — returns a stable
   `400 {"error":"invalid_json"}`. Audit actor/reviewer still come only from the verified
   subject; a rejected body records no event/review.

All D-0006 decisions preserved: Clerk authenticates, Core authorizes; audit actor/reviewer =
verified `sub`; unknown/missing `sidus_role` denied; no content/OCR/ingestion added; no real
Clerk keys in tracked files.

## JSON strict-parsing hardening (final fix, review follow-up)

`decodeStrict` (`services/core/internal/contentsource/handlers.go`) previously called
`json.Decoder.Decode` exactly once with `DisallowUnknownFields()`. `Decode` reads only one
JSON value off the stream and leaves any remaining bytes unread — so a body containing a
valid JSON value followed by a second valid JSON value (e.g. two concatenated objects) passed
silently, with only the first value applied. This let a caller smuggle a second JSON value
(e.g. `{"reviewerId":"..."}`) past `DisallowUnknownFields` undetected.

Fixed: after the first successful decode, `decodeStrict` calls `dec.Decode(new(json.RawMessage))`
again. On a well-formed single-value body this returns `io.EOF` (decoder correctly finds
nothing else). Any other outcome — a second valid JSON value, or trailing non-whitespace
junk — means the body carried more than one value, and the request is rejected with the same
stable `400 {"error":"invalid_json"}` used for every other malformed-body case (no decoder
internals leaked). Trailing whitespace after the single value is accepted, matching normal
JSON-body expectations. Applies uniformly to create/update/approve/reject, since all four
route through the shared `decodeStrict` helper.

New tests in `handlers_test.go`: `TestCreate_RejectsTrailingJSONValue`,
`TestUpdate_RejectsTrailingJunk`, `TestReject_RejectsTrailingJSONValue`,
`TestApprove_TrailingWhitespace_Accepted`,
`TestUpdate_RejectsUnknownActorId_WithTrailingWhitespace` — cover trailing second JSON value
rejected, trailing junk rejected, trailing whitespace accepted, and no event/review/source
mutation on any rejected trailing-body request. Prior legacy `actorId`/`reviewerId` unknown-field
tests (`TestUpdate_RejectsUnknownActorId`, `TestApprove_RejectsUnknownReviewerId`,
`TestReject_RejectsUnknownReviewerId`, `TestCreate_RejectsUnknownField`) remain green, unmodified.

No auth roles, Clerk configuration, routes, migrations, contracts, or content behavior changed.

## Decisions / assumptions

- **Clerk Go SDK pinned to v2.5.0.** v2.6.0/v2.7.0 require Go ≥1.24; the repo (and the
  `golang:1.22-alpine` toolchain used everywhere) is Go 1.22. v2.5.0 has the identical
  verification API (`jwt.Decode`/`GetJSONWebKey`/`Verify`, `AuthorizedPartyHandler`,
  `CustomClaimsConstructor`, `SessionClaims.Custom`). Revisit if/when the toolchain moves to
  ≥1.24.
- **Issuer pinned explicitly** in addition to the SDK's built-in Clerk-issuer format check,
  so tokens from any other Clerk instance are rejected.
- **JWK TTL cache** (default 1h) balances "no Backend API call per request" against picking up
  key rotation; only an unknown/expired-cache `kid` triggers a fetch.
- **Read permission granted to editor/reviewer/admin, denied to learner** — "learner: no
  content-source access" means learner cannot even list/get sources.
- **Fail closed:** content-source routes never mount without a verifier, so a
  misconfiguration cannot expose them unauthenticated.
- **`actorId`/`reviewerId` removed from request bodies.** The obsolete
  `TestUpdate_MissingActorID_Returns400` was replaced by token-based auth tests; identity now
  comes only from the verified subject (`TestUpdate_ActorFromVerifiedSubject`,
  `TestReject_ReviewerFromVerifiedSubject`).
- **Next.js 16 `proxy.ts`** is the current middleware entrypoint; `<Show when=...>` replaced
  `SignedIn`/`SignedOut` in Clerk v7.
- **`apps/web/.env.local` dummy keys** are needed only so `next build`/typecheck run offline;
  they are gitignored and non-functional. Replace with real keys for a working sign-in flow.
